# gdzUNIIT

Веб платформа для публикации и поиска ответов на учебные задания (ГДЗ), сгруппированных по предметам и учебным вариантам: с регистрацией, ролевой моделью доступа, модерацией контента и комментариями.

## О проекте

Готовые ответы на задания обычно разбросаны по разным источникам и не привязаны к конкретному варианту, из-за чего студенту приходится вручную искать нужный среди множества похожих. gdzUNIIT решает эту проблему: при регистрации пользователь указывает свой вариант, и система показывает ему только релевантные ответы — те, что предназначены именно для его варианта, плюс общие ответы, помеченные как доступные для всех вариантов (`isForAllVariants`).


## Функционал

- Регистрация и аутентификация пользователей через Spring Security, выбор учебного варианта при регистрации
- Каталог предметов с карточками-обложками, переход к списку ответов по предмету
- Доступ к ответу проверяется на совпадение варианта пользователя с вариантом ответа (или флаг "для всех вариантов")
- Контент ответов хранится в Markdown и рендерится в HTML на сервере (CommonMark) при сохранении
- Комментарии к ответам с относительным временем публикации ("N минут назад", "только что")
- Загрузка изображений: аватары пользователей, обложки предметов
- Панель модерации с ролевым доступом (USER/ADMIN): список всех пользователей, просмотр и редактирование чужого профиля, управление статусом аккаунта
- Кэширование часто запрашиваемых данных
- Централизованная обработка ошибок через кастомные исключения и страницы 403/404/500
- Черновая интеграция с DonationAlerts API (OAuth-авторизация для приёма донатов) — не доведена до продакшена

## Стек

- **Backend:** Java 17, Spring Boot 3.2 (Web, Security, Data JPA, Mail)
- **БД:** PostgreSQL, Hibernate / JPA
- **Шаблонизатор:** Thymeleaf + Spring Security Thymeleaf Extras
- **Сборка:** Maven
- **Тестирование:** JUnit, Spring Security Test, Testcontainers (PostgreSQL)
- **Прочее:** Lombok, CommonMark (Markdown → HTML), Apache HttpClient, Bootstrap, Font Awesome

## Архитектура

Слоистая архитектура, стандартная для Spring-приложений:

```
Controller → Service (interface + impl) → Repository (Spring Data JPA) → Entity
```

**Доменная модель:**

- `User` — пользователь, реализует `UserDetails` для прямой интеграции со Spring Security; связан с `Variant` (свой вариант), множеством `Role` и списком `Comment`
- `Role` — роль доступа (USER / ADMIN), реализует `GrantedAuthority`
- `Subject` — учебный предмет, содержит список связанных `Answer`
- `Answer` — ответ на задание: заголовок, контент в Markdown, отрендеренный HTML, привязка к предмету и варианту, флаг доступности для всех вариантов, список комментариев
- `Variant` — учебный вариант, к которому привязываются и пользователи, и ответы
- `Comment` — комментарий к ответу с автором и временем публикации

**Обработка ошибок:** доменные исключения (`UserNotFoundException`, `NoAnswersException`, `NoCommentsException`, `VariantNotFoundedException`, `NoSuchSubjectException`) перехватываются контроллерами и центральным `ErrorHandler`, который отдаёт соответствующие страницы 403/404/500.

**Авторизация:** доступ к разделам настраивается в `WebSecurityConfig` через `authorizeRequests` — публичные страницы (главная, регистрация, статика), страницы только для авторизованных (`/answers/**`) и страницы только для администраторов. Проверка пароля вынесена в собственный `CustomAuthenticationProvider`, а не в стандартный `DaoAuthenticationProvider`.

## Запуск локально

1. Поднять PostgreSQL и создать базу `gdz`.
2. Указать параметры подключения в `src/main/resources/application.properties` (`spring.datasource.url`, `spring.datasource.username`, `spring.datasource.password`).
3. Собрать и запустить:

```bash
./mvnw spring-boot:run
```

Приложение поднимется на `http://localhost:8080`.

## Known issues / технический долг

Проект учебный, и часть решений сознательно требует доработки до продакшен-уровня:

- **Хранение паролей в открытом виде.** `CustomAuthenticationProvider` сравнивает пароли как plain text (`user.getPassword().equals(password)`); `BCryptPasswordEncoder` реализован, но закомментирован и не подключён к `AuthenticationManager`. Для продакшена пароли должны хешироваться при регистрации и проверяться через `PasswordEncoder`.
- **Секреты в коде.** `DonationAlertsAPI` и `application.properties` содержат тестовые `clientId`/`clientSecret`/`accessToken` напрямую в исходниках — в реальном проекте это должно уходить в переменные окружения или секрет-хранилище и не попадать в git.
- **OAuth-флоу DonationAlerts не завершён.** `DonationAlertsController.getCode` принимает код авторизации, но не обменивает его на access token — интеграция приёма донатов на уровне черновика.
- Часть закомментированного кода (поля даты регистрации/активности у `User`, неиспользуемые бины в `WebSecurityConfig`) оставлена как заготовка для дальнейшего развития, не удалена намеренно на момент последнего коммита.

## Статус

Основная функциональность — каталог, фильтрация ответов по варианту, аутентификация, ролевая модель, модерация, комментарии — реализована и протестирована вручную. Интеграция с DonationAlerts не доведена до продакшена и отмечена как известное ограничение.
